# XNATWL Efficiency Review

Efficiency / parity review of the hand-written C# port (`XNATWL/XNATWL`) cross-referenced against the
original Java TWL source (`twl/src/de/matthiasmann/twl`). Generated by a multi-agent review sweep across
all 9 subsystems; every High/Critical finding was adversarially re-verified against the Java source to
confirm both that the inefficiency is real **and** that the proposed fix preserves observable parity
with Java.

**Parity is the hard constraint.** Findings are classified:
- `perf` — real C# inefficiency; fix preserves Java-equivalent behavior.
- `parity-perf` — C# is slower than Java because the port diverged from Java's algorithm/data structure; the fix is "match Java".
- `parity-bug` — C# behaves *differently* from Java (a correctness divergence) noticed during review.

20 findings total. None are "Critical" by the data-loss/unbounded-resource bar, but several `parity-bug`
items are crash-or-wrong-result divergences and should be treated as the top priority.

# Part 1 — Renderer, Core Widget, Utils, Misc widgets

## High

### P1-1. Per-glyph `Texture2D` defeats all text batching
`Renderer/XNA/BitmapFont.cs:218-238` — *parity-perf, hot path*

- **Slow:** Font load reads back each glyph from the GPU (`GetData`), runs a Black→Transparent CPU scan,
  and allocates a fresh `Texture2D` + `XNATexture` per glyph. At draw time every character samples a
  different texture, forcing a texture swap per glyph and killing SpriteBatch batching even in `Immediate`
  mode. Load is O(numGlyphs) readbacks + O(glyphArea) scans.
- **Java:** Keeps ONE atlas texture; each `Glyph` is a `TextureAreaBase` holding only sub-rect coords.
  Binds the texture once, emits all glyphs as quads. No per-glyph texture, no per-glyph readback.
- **Fix:** Construct each `Glyph` as a sub-rect of the shared `_texture` (the source-rect path at
  `TextureAreaBase.cs:114` already supports this). Drop the per-glyph `GetData` / recolor loop /
  `new Texture2D` / `new XNATexture`.
- **⚠ Parity:** The Black→Transparent recolor is C#-only (Java relies on `Format.ALPHA`). Apply it ONCE
  to the whole atlas at load to keep byte-identical output.
- **⚠ Parity:** C#-only `CacheBDrawText` (line 628) and `CacheBDrawMultiLineText` (line 790) read
  per-glyph `g.ColorData`. Deleting the readback breaks them unless you keep ONE whole-atlas `Color[]`
  and index it with atlas coords `(_tx0+i) + (_ty0+j)*atlasWidth`.

### P1-2. Forced `GC.Collect()` on every disposing frame
`Renderer/XNA/DeferredDisposer.cs:57-60` — *parity-perf, hot path*

- **Slow:** `Update()` runs once per `EndRendering()` (`XNARenderer.cs:334`). Whenever the dispose queue
  is non-empty it calls `System.GC.Collect()` — a full blocking GC. Font caches retire routinely as
  strings change, so this fires continuously, causing frame hitches worse than the garbage it reclaims.
- **Java:** LWJGL `endRendering()` does only `renderSWCursor()` + `revertGLState()`. No deferred disposer,
  no `System.gc()`. Disposal is pure `glDeleteTextures`.
- **Fix:** Delete the `if (disposedNow > 0) { System.GC.Collect(); }` block. The existing
  `disposed.Dispose()` loop already releases the unmanaged GPU handles.

### P1-3. `_popups[menu]` throws where Java returns null
`MenuManager.cs:64, 174` — *parity-bug*

- **Broken:** `IsSubMenuOpen` (64) and `GetPopupForMenu` (174) use the `Dictionary` indexer →
  `KeyNotFoundException` on a missing key. Both are written to handle null (the latter is *documented*
  to return null). `OpenSubMenu` was ported correctly with `ContainsKey`; these two were missed.
- **Java:** `HashMap.get` returns null; `isSubMenuOpen` null-checks, `getPopupForMenu` returns it.
- **Fix:**
  - `IsSubMenuOpen`: `if (_popups.TryGetValue(menu, out Widget popup) && popup != null) { return popup.GetParent() == this; } return false;`
  - `GetPopupForMenu`: `return _popups.TryGetValue(menu, out Widget p) ? p : null;`
- **⚠ Parity:** Java uses `IdentityHashMap` (reference equality); C# `Dictionary` uses default equality.
  Separate latent issue, not fixed by `TryGetValue`.

## Medium

### P1-4. `CascadedHashMap` does 2-3 probes per level where Java does one
`Utils/CascadedHashMap.cs:96-108` (getEntry), `80-94` (PutCascadingEntry) — *parity-perf, hot path*

- **Slow:** `getEntry` does `ContainsKey` + `map[key] != null` + `map[key]` — up to 3 probes per hit.
  This is the theme-resolution primitive (`ThemeInfoImpl.GetChildThemeImpl`, `ParameterMapImpl.GetParameterValue`).
- **Java:** One `HashEntry.get` per fallback level.
- **Fix:**
  - `getEntry`: `if (map.TryGetValue(key, out var v) && v != null) { return v; }`
  - `PutCascadingEntry`: `if (this.TryGetValue(key, out var oldValue)) { this[key] = value; return oldValue; } else { object cascadedEntry = CascadingEntry(key); this[key] = value; return cascadedEntry; }`
- **⚠ Parity:** Keep the `&& v != null` guard — it preserves current C# behavior. (See parity-bug note below.)

### P1-5. `CacheBDrawMultiLineText` is O(n²) in allocation and measurement
`Renderer/XNA/BitmapFont.cs:699-744` — *perf*

- **Slow:** Wrap loop does `strWithinBounds.Substring(1)` one char at a time (a new string per char) and
  `ComputeTextWidth(line + strWithinBounds[0], ...)` re-measuring the whole line each char. Both O(n²).
- **Java:** No width-wrapping in the font-cache path; splits on `\n` via `TextUtil.indexOf`, measures once per line.
- **Fix:** Index-walk `[start,end)` with a `StringBuilder` + running `lineWidth`. Per char: skip `\r`; on
  `\n` flush+reset; else `adv = GetGlyph(ch) is Glyph g ? g.XAdvance : (ch==' ' ? _spaceWidth : 0)`; if
  `lineWidth + adv > maxLineWidth` flush, reset, retry char on the empty line; else append, `lineWidth += adv`.
  Since `ComputeTextWidth` ignores kerning, incremental width == full remeasure → byte-identical breaks.
  The `if (isNewLine)` branch at 724-727 is dead code.
- *One-time cache bake, not per frame — hence medium.*

### P1-6. `CopySign` allocates 3 `byte[]` via `BitConverter.GetBytes`
`Graph.cs:205-214` — *parity-perf, hot path*

- **Slow:** Three transient `byte[4]` per call, on the paint path for flat/near-flat graph lines
  (`RenderLine` when `delta < EPSILON`).
- **Java:** `Float.floatToRawIntBits` / `intBitsToFloat` — zero allocation.
- **Fix:**
  ```csharp
  int rawMagnitude = BitConverter.SingleToInt32Bits(magnitude);
  int rawSign      = BitConverter.SingleToInt32Bits(sign);
  int rawResult    = rawMagnitude | (rawSign & (1 << 31));
  return BitConverter.Int32BitsToSingle(rawResult);
  ```
  Available on .NET 8. Also drop the stray empty `;` at line 209. Byte-identical.

## Parity bugs noticed (Part 1)

- **P1-3** `MenuManager` indexer throws — primary one, fix included.
- **MenuManager key equality** — Java `IdentityHashMap` vs C# default `Dictionary`. Latent.
- **CascadedHashMap null-value cascade** — Java returns the entry as soon as a key *exists* (a stored
  null stops the cascade); C# skips null-valued entries and keeps cascading. Pre-existing; the perf fix
  neither creates nor cures it. Don't let the `&& v != null` guard mask it.

# Part 2 — TextArea, Tables, Layout, Theme, Models

Most findings here are **parity correctness divergences** with a perf component.

## High — hot path / parity-perf

### P2-1. `BorderLayout` Dictionary indexer throws on absent Location
`BorderLayout.cs:168-196, 223-272, 93, 105` — *parity-bug, hot path*

- **Broken:** `_widgets[Location.X]` throws `KeyNotFoundException` when a location was never added. Any
  BorderLayout missing one of the five locations (the normal case) throws on its first `Layout()` or
  size query — in the per-frame layout/sizing path.
- **Java:** `EnumMap<Location,Widget>` read via `widgets.get(loc)` returns null; surrounding null-checks rely on it.
- **Fix:** Add `private Widget Get(Location loc) { _widgets.TryGetValue(loc, out var w); return w; }`; use
  at 93, 105, 168, 174, 180, 186, 192, 223-272. For `RemoveChild`:
  `_widgets.TryGetValue(location, out var w); _widgets.Remove(location);`.

### P2-2. `Scrollbar.SetValue` throws NRE when no `PositionChanged` subscriber
`Scrollbar.cs:186-189` — *parity-bug, hot path*

- **Broken:** `this.PositionChanged.Invoke(...)` with no null check. An empty C# event is null → NRE on
  any value change of a subscriber-less Scrollbar (scroll, arrows, PgUp/PgDn, wheel, syncModel, handleDrag).
- **Java:** `CallbackSupport.fireCallbacks` guards `if(callbacks != null)`.
- **Fix:** `this.PositionChanged?.Invoke(this, new ScrollbarChangedPositionEventArgs());`

### P2-3. `DefaultTableSelectionModel` materializes the selection `int[]` twice per mutation
`Model/DefaultTableSelectionModel.cs:84-99, 178-234` (+ `TableSingleSelectionModel.cs`) — *parity-perf, hot path*

- **Slow:** Every mutating method reads the `Selection` getter twice; each getter does
  `new int[Cardinality()]` (full BitSet scan) + a `NextSetBit` loop. One `AddSelection` = two full scans
  + two int[] allocations. Fires at event rate during drag/keyboard select. **No consumer reads
  OldSelection/NewSelection** (verified zero reads).
- **Java:** `fireSelectionChange()` is parameterless, zero allocation.
- **Fix:** Drop `int[] oldSelection = this.Selection;` and pass `null, null` to `FireSelectionChange`.
  Apply to **all seven sites** in `DefaultTableSelectionModel.cs` (86/98, 162/169, 180/192, 206/216, 222/233)
  **and** `TableSingleSelectionModel.cs` (90, 108, 117, 127, 151, 172, 181).

### P2-4. `CombinedListModel` binary-search midpoint uses `>> 2` instead of `>> 1`
`Model/CombinedListModel.cs:196` — *parity-bug + parity-perf, hot path* ⚠

- **Broken:** `int mid = (int)((uint)(lo + hi) >> 2);` divides by **4**. The midpoint skews low (can fall
  outside `[lo,hi]`), so bisection converges on the wrong `lo` for any model with >2 sublists → wrong
  `Sublist` → wrong entry from `EntryAt`. Silent wrong result, no exception.
- **Java:** `int mid = (lo+hi) >>> 1;`
- **Fix:** `int mid = (int)((uint)(lo + hi) >> 1);`

## High — parse/load-time correctness

### P2-5. `TextArea` widget/resolver lookup uses Dictionary indexer (throws)
`TextArea.cs:843-846` (also `218`) — *parity-bug*

- **Broken:** `_widgets[we.WidgetName]` then `_widgetResolvers[we.WidgetName]`. The resolver fallback is
  the *expected* case, but the indexer throws first — crashing layout. `UnregisterWidget` at 218 likewise
  throws where Java is a no-op.
- **Java:** `widgets.get(...)` returns null → falls through to resolver; `unregisterWidget` null-checks.
- **Fix:** `_widgets.TryGetValue(we.WidgetName, out var widget); if (widget == null) { _widgetResolvers.TryGetValue(we.WidgetName, out var resolver); if (resolver != null) widget = resolver.ResolveWidget(...); if (widget == null) return; }`. At 218: guard with `TryGetValue`.

### P2-6. `CSSStyle.ParseList` drops font-family entries past the second
`TextAreaModel/CSSStyle.cs:609-647` — *parity-bug*

- **Broken:** Returns at most 2 entries. `font-family: Arial, Helvetica, sans-serif` → `{Arial, Helvetica}`;
  the generic fallback is never tried by `TextArea.SelectFont`. Also allocates a fresh List per recursion.
- **Java:** `new StringList(part, parseList(value, end+1))` — retains all entries in order.
- **Fix:** `var result = ParseList(value, end+1) ?? new List<string>(); result.Insert(0, part); return result;`

### P2-7. `CSSStyle.ParseValueUnits` splits on literal `"\s+"` instead of a whitespace regex
`TextAreaModel/CSSStyle.cs:375` — *parity-bug*

- **Broken:** `value.Split(new string[] { "\\s+" }, ...)` splits on the literal 3-char sequence, which
  never appears in CSS. `margin: 5px 10px 15px 20px` returns one element; 2/3/4-value shorthand is
  unreachable and `ParseValueUnit` throws.
- **Java:** `value.split("\\s+")` — regex, yields 4 parts.
- **Fix:** `System.Text.RegularExpressions.Regex.Split(value, "\\s+")`.

### P2-8. `ThemeManager.ParseList` drops tail and throws NRE
`Theme/ThemeManager.cs:689-701` — *parity-bug* ⚠

- **Broken:** `return new List<string>{ part, ParseList(value, end + 1)[0] };` — keeps at most 2 elements
  and indexes `[0]` on a recursive result that is **null** at list end / for single-element lists → NRE.
  Any font element with a `families` attribute crashes at theme-load time.
- **Java:** `new StringList(part, parseList(value, end+1))` with null base case.
- **Fix:**
  ```csharp
  private static List<string> ParseList(string value, int idx) {
      idx = TextUtil.SkipSpaces(value, idx);
      if (idx >= value.Length) return null;            // Java base case
      int end = TextUtil.IndexOf(value, ',', idx);
      string part = TextUtil.Trim(value, idx, end);
      var result = new List<string> { part };
      var tail = ParseList(value, end + 1);
      if (tail != null) result.AddRange(tail);
      return result;
  }
  ```

### P2-9. `ImageManager` indexes `SortedDictionary` directly, throwing instead of the friendly theme error
`Theme/ImageManager.cs:114` (also `130`, `156`, `293`) — *parity-bug*

- **Broken:** `Image img = _images[reference]; if (img == null) throw ...` — the indexer throws on a missing
  key, so the `img == null` branch is dead code; a theme author gets an opaque `KeyNotFoundException`.
- **Java:** `images.get(ref)` (TreeMap) returns null → friendly `xmlp.error(...)` runs.
- **Fix:** `GetReferencedImage`/`GetReferencedCursor`/`ParseCursor`: `_images.TryGetValue(reference, out var img);`
  then keep the existing `if (img == null) throw`.
- **⚠ `GetCursor` (156) is DIFFERENT** — Java returns **null** for a miss; do **not** add a throw:
  `_cursors.TryGetValue(name, out var cursor); return UnwrapCursor(cursor);`.

### P2-10. `ColumnLayout.Columns.Equals` uses reference equality on the names array
`ColumnLayout.cs:208` — *parity-bug*

- **Broken:** `this._hashCode == other._hashCode && this._names == other._names;` — reference equality.
  Names are cloned per instance (185), so two `Columns` from identical name lists never compare equal →
  defeats the `Dictionary<Columns,Columns>` dedup cache and prefix-merge sharing.
- **Java:** `Arrays.equals(this.names, other.names)`.
- **Fix:** keep the hash short-circuit, then element-wise:
  ```csharp
  if (other._names.Length != this._names.Length) return false;
  for (int i = 0; i < this._names.Length; i++)
      if (!string.Equals(this._names[i], other._names[i])) return false;
  return true;
  ```

### P2-11. `FileTable` sort order (Ascending/Descending) is never applied
`FileTable.cs:313-319` — *parity-bug*

- **Broken:** `Array.Sort(entries, numFolders, entries.Length, SortColumn_Comparer(_sortColumn))` folds in
  only `_sortColumn`. `_sortOrder` only drives the arrow glyph → descending flips the arrow but rows stay
  ascending. The port dropped `SortOrder.map()`.
- **Java:** `Arrays.sort(..., sortOrder.map(sortColumn.comparator))`; `DESCENDING.map(c)` returns
  `Collections.reverseOrder(c)`.
- **Fix:** wrap the comparer when descending with a `ReverseComparer` whose `Compare(a,b) => _c.Compare(b, a)`
  (swap operands, not negation). Do **not** touch the arrow logic at 342-355.

## Medium

### P2-12. `TreeTable.GetOrCreateNodeState` does two Dictionary lookups
`TreeTable.cs:287-299` — *parity-perf, hot path*

- **Slow:** `_states.ContainsKey(node)` then `_states[node]` — two probes per visible non-leaf cell per
  paint frame (via `GetCellRenderer` at 407).
- **Java:** `HashEntry.get(...)` — single probe + null-check.
- **Fix:** `if (!this._states.TryGetValue(node, out var ns)) ns = CreateNodeState(node); return ns;`
  (`_states` never stores null.)

### P2-13. `EditField.PasswordMasker.Value` builds the masked string with O(n²) concatenation
`EditField.cs:1631-1642` — *parity-perf, hot path*

- **Slow:** `outputValue += this.CharAt(i)` reallocates+copies each iteration → O(n²) + O(n) garbage.
  Materialized on every `UpdateTextDisplay` (each keystroke) and `ComputeCursorPosFromX` (mouse).
- **Java:** `PasswordMasker` is a lazy `CharSequence`, never materialized.
- **Fix:** `public string Value => new string(_maskingChar, this.Length);`

### P2-14. `ParameterListImpl.GetParameterValue<T>` casts to T before the type check

- **Fix:** optional — delete the unused generic overload (and its `ParameterList.cs:151` interface decl).
